claude-config-manager

A Claude Code plugin and npm package for managing configurations, plugins, MCP servers, profiles, skills, and custom commands through a unified interface.

Version 1.0.0 Architecture Spec 2026-03-25 Status: Draft

Tech Stack

The project is organized as a Turborepo monorepo with five packages, each with clearly scoped responsibilities and minimal coupling. All packages share a common TypeScript configuration base and are built with the Turbo pipeline.

Utility Modules

claude-home.ts Resolves the ~/.claude/ base directory cross-platform. Respects the CLAUDE_CONFIG_DIR environment variable override. All other modules reference paths through this utility rather than hardcoding paths.
file-ops.ts Provides atomic JSON read/write operations. Writes go to a temporary file first, then are renamed into place (atomic on most filesystems). Includes a simple advisory lock mechanism to coordinate concurrent writes from multiple entry points.
marketplace.ts Wraps git operations for marketplace management. Handles git clone for initial install, git pull for updates, and provides commit-hash-based version tracking.

Every entry point follows the same data flow pattern: user intent enters at the boundary (CLI command, HTTP request, MCP tool call, or plugin hook), is translated into a @ccm/core manager method call, which performs atomic filesystem operations on ~/.claude/ and returns typed results.

Profile System

Profiles provide named snapshots of Claude Code configuration. A profile captures: settings (model, permissions, hooks), enabled plugins and their configuration, and MCP server registrations. Users can switch between profiles to rapidly change their working context.

Profile Storage

~/.claude/plugins/profiles/

├── abc123.json                  # Profile "development"
├── def456.json                  # Profile "production-review"
└── ghi789.json                  # Profile "minimal"

Profile Schema

interface Profile {
  id: string;                    // UUID v4
  name: string;                  // Human-readable name
  description?: string;          // Optional description
  createdAt: string;             // ISO 8601 timestamp
  updatedAt: string;             // ISO 8601 timestamp
  settings: {
    model?: string;              // e.g., "claude-sonnet-4-20250514"
    permissions?: PermissionSet;
    hooks?: HookConfig[];
    env?: Record<string, string>;
  };
  enabledPlugins: string[];      // Plugin IDs
  mcpServers: Record<string, McpServerConfig>;
}
Profile Isolation Profiles do not copy plugin files — they only reference plugin IDs. When switching profiles, the system enables/disables plugins and updates MCP server configs, but the actual plugin cache remains shared.

Active Profile Tracking

The active.json file is a simple pointer that stores the ID and name of the currently active profile. If no profile is active, this file either does not exist or contains null. The dashboard header displays the active profile name, and the CLI shows it with an indicator in claude-config list profiles.

Security Considerations

Credential Exclusion

The .credentials.json file is never included in export operations by default. An explicit --include-secrets flag is required to include credentials, and the CLI/dashboard display a prominent warning before proceeding.

Environment Variable Masking

MCP server environment variables (which may contain API keys) are masked in dashboard display using the pattern sk-...XXXX. Full values are only shown after explicit user action (click-to-reveal).

Path Traversal Prevention

All file operations in file-ops.ts validate that resolved paths remain within the ~/.claude/ directory tree. Paths containing .. segments or symlinks pointing outside the boundary are rejected.

Import Schema Validation

Imported configuration JSON is validated against the expected schema before any changes are applied. Invalid structures, unexpected fields, and type mismatches are rejected with descriptive error messages.

Concurrency Model

Multiple entry points (CLI, dashboard, MCP server, plugin hook) may attempt to read or write configuration files simultaneously. The concurrency model is designed to prevent data corruption while remaining simple and dependency-free.

Atomic File Operations

All write operations in file-ops.ts follow the write-to-temp-then-rename pattern:

  1. Write the new content to a temporary file in the same directory (e.g., settings.json.tmp.{random})
  2. Call fsync to flush to disk
  3. Rename the temp file to the target path (atomic on POSIX; best-effort on Windows)
  4. If rename fails, clean up the temp file and throw

Advisory Locking

For operations that require read-modify-write cycles (e.g., adding a plugin to installed_plugins.json), an advisory lock file is used: 

async function withLock<T>(lockPath: string, fn: () => Promise<T>): Promise<T> {
  const lockFile = lockPath + '.lock';
  await acquireLock(lockFile, { retries: 5, retryDelay: 100 });
  try {
    return await fn();
  } finally {
    await releaseLock(lockFile);
  }
}
Windows Compatibility Note On Windows, file rename is not guaranteed atomic if the target exists. The implementation uses fs.rename which on Windows performs a non-atomic replace. For most use cases this is acceptable, as simultaneous writes from multiple entry points are rare in practice. The advisory lock provides the primary concurrency protection.

Key Design Decisions

1. Monorepo with Turborepo

Chosen over a single-package architecture to enforce clear separation of concerns. The CLI, dashboard, and MCP server have distinct deployment targets and dependency trees. Turborepo provides efficient caching and parallel builds without the complexity of Nx or Lerna.

2. Pure Core Package

@ccm/core has zero UI dependencies and no framework-specific code. This ensures it can be consumed by any entry point — Node.js CLI, Next.js server components, MCP server process, or directly by the Claude Code plugin runtime — without dependency conflicts.

3. stdio Transport for MCP Server

The MCP server uses stdio rather than HTTP/SSE transport. This is the standard transport for Claude Code MCP servers and keeps the server lightweight — no HTTP server, no port management, no CORS. The process is spawned by Claude Code and communicates via stdin/stdout JSON-RPC messages.

4. Profiles in plugins/profiles/ Subdirectory

Profile data is stored under ~/.claude/plugins/profiles/ rather than directly in ~/.claude/. This avoids adding new files to the root of the Claude configuration directory, which could interfere with Claude Code's own file discovery. The plugins/ prefix signals that this data belongs to the config manager plugin.

5. Thin API Routes

Dashboard API routes contain no business logic. They parse request parameters, instantiate the appropriate @ccm/core manager, call the corresponding method, and serialize the result. This ensures that behavior is identical regardless of whether the user interacts via the dashboard, CLI, or MCP server.

6. Git-Based Plugin Distribution

Plugins are distributed as git repositories rather than npm packages. This allows for simpler publishing (just push to a repo), version tracking via commit hashes, and easy updates via git pull. Marketplace repositories serve as curated indexes pointing to individual plugin repos.

Error Handling Strategy

Error Classification

All errors thrown by @ccm/core are instances of typed error classes that extend a base CcmError: 

class CcmError extends Error {
  constructor(
    message: string,
    public readonly code: string,
    public readonly recoverable: boolean = false
  ) { super(message); }
}

class FileNotFoundError extends CcmError { ... }
class ValidationError extends CcmError { ... }
class LockAcquisitionError extends CcmError { ... }
class PluginInstallError extends CcmError { ... }
class GitOperationError extends CcmError { ... }

Error Propagation by Entry Point

Entry PointError Handling
CLICatches CcmError, formats with chalk (red for fatal, yellow for recoverable), exits with appropriate code
Dashboard APICatches CcmError, returns JSON { error: { code, message } } with appropriate HTTP status
MCP ServerCatches CcmError, returns MCP error response with code and message in the tool result
Plugin HookLogs to Claude Code plugin log; non-fatal errors are swallowed to avoid disrupting the host
Unrecoverable Errors If the ~/.claude/ directory is missing, inaccessible, or has incorrect permissions, core managers throw a non-recoverable FileNotFoundError with a descriptive message guiding the user to check their Claude Code installation.

Future Considerations

Cloud-based profile synchronization allowing users to share profiles across machines. Would require an authentication layer and a storage backend (likely a simple JSON blob store).

Phase 2

Plugin Dependency Resolution

Currently plugins are independent. Future versions may support declaring dependencies between plugins, requiring a topological sort during enable/disable operations.

Phase 3

Marketplace Web UI

A browsable marketplace interface within the dashboard, showing plugin ratings, descriptions, and one-click install. Requires a marketplace index API.

Phase 3

Configuration Diff View

Visual diff between profiles or between current config and a profile, allowing users to understand what will change before switching.
