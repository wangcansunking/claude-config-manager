Problem Statement

Claude Code power users manage complex configurations across multiple machines: plugins, MCP servers, custom slash commands, hooks, model settings, and environment variables. Today, this configuration lives in scattered JSON files with no visual tooling, no portability between machines, and no way to quickly switch contexts (e.g., "Work" vs. "Personal" vs. "Open Source").

claude-config-manager solves this by providing a unified web dashboard, a set of MCP tools for AI-assisted config management, a CLI for scripting, and a profile system for portable, switchable configuration snapshots.

Target Audience

Power Users

Developers who use Claude Code daily with 5+ plugins and multiple MCP servers. They need visibility and control.

Multi-Machine Users

Users who work across laptop, desktop, and CI environments. They want portable configs that follow them.

Team Leads

Engineering leads who want to share standardized Claude Code setups with their teams.

Architecture

Monorepo Packages

The project is structured as a Turborepo monorepo with five packages:

@ccm/core
Configuration read/write engine, file watchers, validation
@ccm/dashboard
Next.js web UI (Notion/Linear style, dark theme)
@ccm/mcp
MCP server exposing 19 tools to Claude
@ccm/cli
CLI binary (claude-config)
@ccm/types
Shared TypeScript types & Zod schemas
|
Claude Code Config Files
~/.claude/ — settings.json, plugins/installed_plugins.json, plugins/profiles/

Dual-Mode Operation

Global npm Package

npm install -g claude-config
claude-config start

Install globally, launch the dashboard from any terminal. Useful for users who want a standalone GUI.

Claude Code Plugin

claude plugin install claude-config-manager

Registers as an MCP server within Claude Code. All 19 tools become available to the AI assistant directly.

User Stories

Power User
As a developer using 8 plugins and 3 MCP servers, I want a visual dashboard so I can see everything at a glance instead of editing JSON files.
Multi-Machine User
As someone who works on both a laptop and a desktop, I want to export my configuration to a file and import it on the other machine so I have the same setup everywhere.
Team Lead
As a team lead, I want to create a "Team Standard" profile and share it with my team so everyone has the same MCP servers and recommended plugins.
Context Switcher
As a developer who consults for multiple clients, I want named profiles (e.g., "Client A", "Client B") so I can switch configurations instantly without manual editing.
AI-First User
As someone who prefers working through Claude, I want MCP tools so I can ask Claude to "add the GitHub MCP server" or "switch to my Work profile" without leaving the conversation.
Automation Engineer
As a CI/CD engineer, I want CLI commands so I can script profile activation in my setup scripts and dotfile repos.
Security-Conscious User
As a security-minded developer, I want credentials excluded from profile exports by default so I do not accidentally share API keys.
Plugin Explorer
As a user discovering new plugins, I want to browse a marketplace-style list and see each plugin's skills and agents before installing.

CLI Examples

# Launch dashboard on custom port
claude-config start --port 4200

# List all plugins as JSON
claude-config list --plugins --json

# Create a profile from current state
claude-config profile create "Work" --description "Daily work config"

# Switch profiles
claude-config profile activate "Personal"

# Export for sharing
claude-config export "Work" --format yaml --output ./work-profile.yaml

# Dry-run import to see what would change
claude-config import ./teammate-profile.json --dry-run

# Full import with replace strategy
claude-config import ./teammate-profile.json --replace

4. Profiles System

Overview

Profiles are named configuration snapshots that capture the complete state of a Claude Code setup. They enable instant context-switching and portable configuration sharing.

What a Profile Contains

Plugins

  • Installed plugin list
  • Enable/disable state for each
  • Version pinning (optional)

MCP Servers

  • Server name and command
  • Arguments and env vars
  • Credentials excluded by default

Settings

  • Model preference
  • Environment variables
  • System prompt overrides

Hooks

  • Pre/post hooks for events
  • Custom scripts
  • Lifecycle callbacks

Commands

  • Custom slash commands
  • Command arguments schema
  • Linked skills

Permissions

  • Allowed file paths
  • Network permissions
  • Tool restrictions

Import Strategies

Merge (Default)

Keep existing configuration and add new items from the imported profile. Conflicts are resolved by keeping the local value. Best for: adding a teammate's plugins to your existing setup.

Replace

Completely replace the current configuration with the imported profile. Existing items not in the import are removed. Best for: setting up a fresh machine to match another exactly.

Security

Credential Handling

By default, profile exports exclude all credentials and secrets. This includes API keys in environment variables, authentication tokens in MCP server configs, and any value matching common secret patterns (keys containing SECRET, TOKEN, KEY, PASSWORD, CREDENTIAL).

Users can opt-in to include credentials via a dashboard toggle or the --include-secrets CLI flag, but will see a warning confirming the security implications.

User Flows

Flow 1: First-Time Setup

Install package → Run claude-config start → Dashboard opens → Auto-detects existing config → Overview page

Flow 2: Profile Export & Import

Open Profiles page → Click Export → Choose format (JSON/YAML) → Download file
Open Import page → Drop file → Preview diff → Choose strategy → Apply

Flow 3: AI-Assisted Config via MCP

User: "Add the GitHub MCP server" → Claude calls ccm_add_mcp_server → Config updated → Dashboard reflects change

Flow 4: Context Switching

User: "Switch to Personal profile" → Claude calls ccm_activate_profile → Plugins toggled → MCP servers reconfigured → Settings applied

Flow 5: Plugin Discovery & Install

Open Plugins page → Click Marketplace tab → Search / browse → Click Install → Review skills in detail panel → Confirm

Risks & Mitigations

RiskImpactLikelihoodMitigation
Claude Code config format changes High Medium Abstract config access behind versioned adapters in @ccm/core. Add schema migration system.
Credential leakage in profile exports High Low Credentials excluded by default. Pattern-based detection. Explicit opt-in with warning.
Profile import corrupts config High Low Automatic backup before import. Dry-run mode. Zod validation on all inputs.
Dashboard performance with many plugins Medium Medium Virtual scrolling for lists. Lazy-load detail panel content. Cache config reads.
MCP protocol breaking changes Medium Medium Pin MCP SDK version. Add protocol version negotiation. Maintain compatibility matrix.
Low adoption due to CLI-only alternatives Medium Medium Focus on visual diff/preview features that CLI cannot match. Emphasize MCP integration as unique differentiator.

A. Profile Export Schema (v1.0)

The complete JSON schema for exported profile files:

{ "version": "1.0", // Schema version for forward compat "name": "Work", // Human-readable profile name "description": "Daily work configuration", // Optional description "createdAt": "2026-03-25T10:30:00Z", // ISO 8601 timestamp "exportedAt": "2026-03-25T14:22:00Z", // When this file was exported "machine": "MacBook-Pro.local", // Source machine hostname "ccmVersion": "1.0.0-draft", // claude-config-manager version "plugins": { "installed": [ // List of installed plugins { "name": "todoist-plugin", "version": "1.2.0", "source": "npm" // "npm" | "github" | "local" }, { "name": "git-assistant", "version": "2.0.1", "source": "npm" } ], "enabled": { // Enable/disable map "todoist-plugin": true, "git-assistant": true, "code-review": false } }, "mcpServers": { // MCP server configurations "filesystem": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user"] }, "github": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-github"], "env": {} // Credentials stripped } }, "settings": { "model": "claude-sonnet-4-20250514", "maxTokens": 16384, "temperature": 0.7, "systemPrompt": "You are a helpful assistant...", "env": { // Non-sensitive env vars only "NODE_ENV": "development", "LOG_LEVEL": "info" } }, "hooks": { "preCommit": ["npm run lint", "npm run format"], "postResponse": ["node scripts/log-response.js"] }, "commands": [ // Custom slash commands { "name": "deploy", "description": "Run deployment workflow", "script": "./scripts/deploy.sh", "args": [ { "name": "env", "type": "string", "required": true } ] } ], "permissions": { // Optional permissions "allowedPaths": ["/home/user/projects"], "networkAccess": true, "shellExecution": true } }
